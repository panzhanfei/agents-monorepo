# 客户侧导引：这套 Agent 是做什么的

面向 **第一次接触** 本编排系统的业务方与群管理员：用少量概念把「谁在干什么」说清楚；**具体飞书句式**以 [FEISHU_COMMANDS.md](./FEISHU_COMMANDS.md) 为准。

---

## 1. 一句话

你们在群里用自然话描述想做的事；**orchestrator（编排）** 把话拆成步骤，依次交给 **5 个专属 Agent**，从需求整理到可发布结果（中间包含写代码、审查、测试、发包/巡检，视你们配置而定）。

---

## 2. 五个专属 Agent（加上编排共 6 个应用）

| 角色 | 您可理解成 |
|------|------------|
| **编排（orchestrator）** | 总指挥：识别您在飞书里发的意图、管任务状态、把活派给下游并把进度/结果写回群里。 |
| **需求分析（requirements-agent）** | 把「想法」整理成 **PRD、验收标准、风险**；**不**直接改您业务仓库里的代码。 |
| **编码（coding-agent）** | 在已绑定的工作区里 **改代码**（分支与 Git 策略以部署方约定为准）。 |
| **审核（review-agent）** | 跑 **工具链门禁**（如 lint/typecheck）并做 **规则/AI 评审**，给出能否合并/继续的建议。 |
| **测试（test-agent）** | 在您的项目里跑 **配置好的全量测试**，汇总报告（多长耗时取决于项目）。 |
| **运维（ops-agent）** | **打包、发布、备份/回滚、只读巡检** 等；涉及生产变更时通常会 **要验证码**（见配置说明）。 |

编排不负责「代替五个 Agent 干具体活」，只负责 **调度与状态**；共享的步骤类型与类型定义在 **`packages/pipeline-core`**，避免各应用各写一套名词。

---

## 3. 推荐第一次怎么玩（顺序）

1. 向部署方确认：**工作区路径**、**测试/发布命令**、**验证码是否开启**。  
2. 若是新项目：按指令表做 **绑定工作区**（常需验证码）。  
3. 发一条 **需求分析**：把背景、用户、要达成的效果说清楚。  
4. 对齐 PRD/验收后，再发 **编码** 任务。  
5. 改完走 **审核** → **全量测试**；通过后再 **发包/一键发布**（按表操作，留意验证码与分步确认）。  
6. 平时可在群里发 **「帮助」** 或 **「新手指引」**，机器人会回一段 **内置导引**（与 [FEISHU_COMMANDS.md §0](./FEISHU_COMMANDS.md) 一致）；发 **「状态」** 可看近期任务。

---

## 4. 和「技术栈」的关系（了解即可）

若部署启用了 **实现面 + 栈画像**（如前端 Next / Vue、后端 Nest），编排会把这类信息带给编码/审核/测试，让机器人和您仓库里的命令对齐；详见 [ARCHITECTURE.md §15](./ARCHITECTURE.md#15-运行时-skill-划分v1)。业务同事 **不必背枚举**，只需在需求里说明 **技术栈与仓库结构**，由实施方映射到配置。
