# 后续规划：小程序、飞书 H5 与可选飞书机器人（阶段三，未定排期）

与 [总览：版本落地顺序](./overview.md) 一致：**当前不设时间表** 做小程序与飞书；待 **Node + Agent** 与 **React（Electrobun）** 稳定后再做。原则不变：**客户仓只在本机 Runner 落盘**；各渠道只连 **Express**，共用 **账号、项目、任务、队列、心跳** 模型。用户不在电脑旁时，用手机 **查状态、审阅、确认/驳回、触发已编排步骤**（能力以实际实现为准）。

## 能力与边界

| 方向 | 说明 |
|------|------|
| **微信小程序** | 经 **微信登录 / code2session** 等与 **产品内用户** 绑定；请求 **同一套 HTTPS API**；UI 侧重 **轻量查询与操作**，复杂编排仍以桌面为主。 |
| **飞书 H5** | 飞书 **应用内网页 / 自建 H5**，用开放平台 **登录与 JSAPI** 换 **产品内 `userId`**，**直接调 Express**；与 React 桌面 **同一 REST 契约**，无需机器人也能闭环。 |
| **飞书机器人（可选、后期）** | **近期不排期**；有余力再评估是否接入（会话内消息、卡片、事件订阅 Webhook 验签等）。若只做 **被动通知**，也可能与 H5 并存：**机器人推送摘要 + 深链打开 H5**。 |
| **能做什么** | 查询任务状态、收摘要与告警、**确认/驳回**、触发「已编排好的」步骤等；小程序侧文档占位 `WECHAT_MINIPROGRAM.md`；若将来启用机器人再维护 `FEISHU_COMMANDS.md`（或与 `CHANNEL_COMMANDS.md` 合并）。 |
| **不能绕开的物理限制** | **写盘、跑 git、跑本机测试** 仍只发生在 **用户电脑上的 Runner**；各渠道 **只到云端**，由云端 **向已在线的 Runner 投递**；**Runner 离线则只能排队或明确提示失败**。 |

## 安全与身份：为何要强绑「桌面是否已登录」

**微信 openId / 飞书侧用户标识** 均须映射到 **同一产品内 `userId`**；对 **会影响工作区** 的操作，建议 **强制下列条件之一（可组合）**：

1. **Electrobun 桌面端已登录** 且 **同设备 Runner 对云端心跳/长连接在线**（云端记录 `userId` + `runnerDeviceId` + 最近心跳）。手机 / 飞书 H5 仅作为 **遥控**，不引入第二条写盘通道。  
2. **设备级令牌**：桌面首次登录后签发 **短期或可调度的 Runner 令牌**，远端触发的任务 **只投递到持有该令牌的 Runner**；令牌吊销即禁止远端写盘类操作。  
3. **人机二次确认**：高危操作可在 **桌面弹窗 / 系统通知** 再确认一次（实现期选型），避免误触。

**落地含义**：服务端在处理 **小程序或飞书 H5（及将来的飞书机器人）** 请求、准备向 Runner 派发 **写盘类任务** 前，**校验该用户名下是否存在已认证且在线的 Runner（或有效设备会话）**；否则返回明确错误（例如「请打开电脑客户端并保持登录」），而不是降级到服务器写盘。

## 与 monorepo 的关系

- **飞书 H5**：与桌面 React **同源 API**；静态资源部署按飞书应用配置即可，**不必** 为 H5 单独写一套入队逻辑。  
- **飞书机器人（若以后做）**：Webhook 路由可放在 `apps/api` 内或后续 `packages/feishu-webhook`。  
- **微信小程序**：独立目录如 `apps/wechat-miniprogram`（或 `miniprogram/`），**仅调云端 API**；契约仍走 `contracts/`。  
- **统一接入层**：Express 收敛 **鉴权、限流、Runner 在线校验、入队**；各渠道只做 **身份换 Token + 调 REST**（机器人额外保留 **验签与消息解析** 一层）。
